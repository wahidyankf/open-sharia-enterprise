# OrganicLever Web — Architecture

**Audience:** Engineers, Technical Product/Project Managers

The OrganicLever web app (`organiclever-web`) is structured around **DDD (Domain-Driven
Design) bounded contexts** (a software architecture approach that groups related code by
business capability). Each bounded context owns its `domain`, `application`,
`infrastructure`, and `presentation` layers — and ESLint enforces the layering rules at
error severity.

## Bounded contexts

| Context           | Responsibility                                                 |
| ----------------- | -------------------------------------------------------------- |
| `journal`         | Append-only event log — system of record for everything logged |
| `routine`         | Workout templates the user authors                             |
| `workout-session` | Active workout FSM (XState v5 state machine)                   |
| `stats`           | Read-only projections and aggregates over journal events       |
| `settings`        | User preferences (dark mode, language, rest timer)             |
| `app-shell`       | Navigation chrome, overlay tree, i18n, `appMachine`            |
| `health`          | Backend health-endpoint client (dormant in v0)                 |
| `landing`         | Marketing landing page at `/`                                  |
| `routing`         | Disabled-route 404 guards (`/login`, `/profile`)               |

See [bounded-context-map.md](./ddd/bounded-context-map.md) for strategic-pattern
relationships and ESLint boundary rules between contexts.

## Project layout

```
apps/organiclever-web/
├── src/
│   ├── app/                        # Next.js App Router (thin page wrappers only)
│   │   ├── app/                    # /app/* routes (home, history, progress, settings, workout)
│   │   └── system/status/be/       # Server-rendered backend diagnostic page
│   ├── contexts/                   # Bounded-context implementations
│   │   ├── app-shell/              # Navigation chrome, i18n, entry-logging overlays
│   │   ├── health/                 # Backend health diagnostic (dormant BE client)
│   │   ├── journal/                # Event log — system of record
│   │   ├── landing/                # Marketing landing page
│   │   ├── routine/                # Workout routine management
│   │   ├── routing/                # 404 guards (disabled routes)
│   │   ├── settings/               # User preferences
│   │   ├── stats/                  # History + progress projections
│   │   └── workout-session/        # Active workout FSM (XState v5)
│   ├── shared/                     # Cross-context utilities
│   │   ├── runtime/                # PgliteService Tag, AppRuntime, shared errors
│   │   └── utils/                  # format-relative-time, fmt
│   └── generated-contracts/        # Auto-generated from OpenAPI spec (gitignored)
└── test/unit/steps/                # Vitest-cucumber step implementations
```

Each bounded context follows the same internal layout:

```
src/contexts/<bc>/
├── domain/           # Pure types, invariants, tagged errors — no IO
├── application/      # Use-cases, XState machines — depends on domain only
├── infrastructure/   # PGlite stores, Effect Layers — depends on domain + application
└── presentation/     # React hooks + components — depends on domain + application
```

## Layer rules

The following import rules are enforced by ESLint `boundaries` at **error** severity.
Violations cause `nx run organiclever-web:lint` to fail.

| Layer            | May import from                               |
| ---------------- | --------------------------------------------- |
| `domain`         | Nothing inside this project                   |
| `application`    | `domain` only                                 |
| `infrastructure` | `domain` + `application` + `@/shared/runtime` |
| `presentation`   | `domain` + `application`                      |

Cross-context coupling is allowed **only** through a context's published barrel:
`application/index.ts` or `presentation/index.ts`. Importing from a sibling context's
internal files (e.g., `@/contexts/journal/infrastructure/journal-store`) is an ESLint
error.
