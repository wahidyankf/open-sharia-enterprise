## FALSE_POSITIVE: content-nesting-accuracy | nist-sp-800-53-rev5.md | MEDIUM-1-POPULATION-B

**Key**: nist-sp-800-53-rev5/MEDIUM-1-POPULATION-B
**Accepted**: 2026-05-14
**Category**: content-nesting-accuracy
**File**: docs/reference/security/frameworks/nist-sp-800-53-rev5.md
**Source Audit**: generated-reports/pdf-to-md**e84ba8_26a05b_3c9d2f**2026-05-14--18-21\_\_audit.md

**Finding**: 192 letter items (`a.`/`b.`/`c.`) at 4-space indent in controls AC-5, AC-7, AC-8, AC-11, and similar controls where the entire control body block uses consistent 4-space indentation.

**Reason**: The PDF itself (verified via `pdftotext -layout`) shows whole-block indentation for these controls. The control identifier (`AC-5`, `AC-7`, etc.) sits at the left margin; the entire body — control name, `Control:`, letter items, `Discussion:`, `Related Controls:`, `Control Enhancements:`, `References:` — is indented as a unit. The `a.`/`b.`/`c.` letter items are at the SAME column as `Control:` and `Discussion:` in the PDF, not more deeply nested. The 4-space MD indentation faithfully reproduces this whole-block PDF layout. Checker misidentified the unit of indentation as the letter items specifically rather than the entire control body block.
